# 全流程使用指南：从零到部署 + QA 反馈闭环

> 本文档说明如何用 model-factory-skills 从零开始，走完「定方向 → 策划数据 → 采集清洗 → 训练 → RL 对齐 → 评测 → 推理优化 → 封 API → 部署 → QA 验证 → 不满足则重训直到通过」的完整闭环。

---

## 整体闭环

```
定方向 → 定需求 → 策划数据 → 采集清洗 → 训练 → RL对齐 → 评测
  ↑                                                          │
  │                    不达标 → 重训（≤2轮）←──────────────────┤
  │                                                          │
  │              达标 → 推理优化 → 封API → 部署 → QA验证 ──→│
  │                                            不通过 ↗      │
  │                                          ────────        │
  │                                         再训练/修复      │
  └──────────────────────────────────────────────────────────┘
                        全部通过 → 上线！
```

核心机制：**两层闸门**（离线评测 + 线上 AB）+ **反馈闭环**（不达标自动回流重训，≤2 轮，第 3 轮上报人工决策）。

## 第二步：按阶段调用 Agent（完整 7 阶段）

### Phase 1：定方向与需求

| 步骤 | 调用 | 做什么 | 产出 | 对应技能 |
|------|------|--------|------|----------|
| 1.1 | `@ceo` | 定垂直场景、选基座（自训/微调/API）、算力投入 | 场景定义 + 技术路线 + 算力预算 | `ceo-model-strategy` |
| 1.2 | `@po` | 写模型能力 PRD：做什么能力、可评测成功标准、边界 | PRD + 数据需求 + 评测目标 | `model-product-prd` |
| 1.3 | CEO 走查 PRD | 质疑方向对不对、基座选对没、评测可量化？ | 通过→继续 / 打回→修改（≤2轮） | — |

**示例对话：**

```
@ceo 我们要做法律合同智能审查，选什么基座，算力预算多少
@po 基于CEO方向，出一份"法律合同条款抽取"的PRD
```

**CEO 走查要点：**
1. 方向一致性 — PRD 是否符合战略方向和当前 OKR？
2. 基座选型 — 自训/微调/基座是否合理？算力预算匹配？
3. 评测标准 — 评测指标是否可量化？
4. 可执行性 — 现有资源下是否可实现？

走查通过 → PO 继续级联下游；打回 → PO 修改（≤2 轮，第 2 轮仍打回则上报用户）。

### Phase 2：数据策略 + 采集清洗

| 步骤 | 调用 | 做什么 | 产出 | 对应技能 |
|------|------|--------|------|----------|
| 2.1 | `@data-strategy` | 定数据来源、标注规范、质量标准、"够不够训练"判断 | 数据需求文档 + 质量标准 | `data-strategy-spec` |
| 2.2 | `@data-engineer` | 采集→清洗→去重→脱敏→版本化→质量校验 | 版本化数据集 + 校验报告 | `data-pipeline-build` |

**示例对话：**

```
@data-strategy 法律合同数据从哪来？够不够训练？标注怎么做？
@data-engineer 按数据策略搭建采集清洗流水线，输出 v1 数据集
```

**数据策略关键决策：**

| 来源 | 优点 | 风险 |
|------|------|------|
| 公开数据 | 免费、快 | 质量参差、同质化 |
| 人工标注 | 质量高 | 慢、贵 |
| 合成数据 | 快、可控 | 可能退化、需校验 |
| 采购 | 快 | 成本、合规 |

> ⚠️ 评测数据必须独立于训练数据，否则评测作弊。合成数据必须校验，避免模型学自己的输出。

**数据流水线五步：**
1. **采** — 按来源清单采集，合规先行（是否可商用、是否含 PII）
2. **洗** — 去重（精确+模糊）、去毒、脱敏、格式归一、长度/质量过滤
3. **版本化** — 版本号+元信息+hash，训练结果可复现
4. **质量校验** — 分布/重复率/噪声率/覆盖率
5. **自动化** — 采→洗→版本→校验串成流水线，更新自动触发

**交接标准：** 数据通过质量校验 → 交 @ml-trainer 训练

---

### Phase 3：训练

| 步骤 | 调用 | 做什么 | 产出 | 对应技能 |
|------|------|--------|------|----------|
| 3.1 | `@ml-trainer` | 设计实验、超参、Scaling 评估、训练监控 | 可复现实验记录 + checkpoint + 日志 | `ml-pretraining-experiment` |

**示例对话：**

```
@ml-trainer 用 dataset-v1 在 Qwen2.5-7B 上做 SFT 实验，记录全链路
```

**训练实验五步：**
1. **实验设计** — 记录实验 ID + 数据版本 + 超参 + 种子 + 目标假设
2. **Scaling 评估** — 算力是硬约束，先小规模验证再 scale up
3. **训练与监控** — loss 曲线、异常止损（发散→停→查根因）
4. **实验追踪** — 全链路记录到 W&B/MLflow
5. **交付评测** — 出 checkpoint + 训练日志，等 @evaluation 的 Go/No-Go

> ⚠️ 训练完说"模型训好了"不算数。@evaluation 说 Go 才算数。

**交接标准：** 模型能跑通基础能力 → 交 @ml-alignment 对齐

### Phase 4：RL 对齐

| 步骤 | 调用 | 做什么 | 产出 | 对应技能 |
|------|------|--------|------|----------|
| 4.1 | `@ml-alignment` | 选对齐方法（SFT→DPO→RLHF）、构建偏好数据、安全红队 | 对齐后模型 + 安全数据集 | `ml-alignment-rlhf` |

**示例对话：**

```
@ml-alignment 基座训完了，做 SFT+DPO 对齐，构建偏好数据，做安全红队
```

**对齐方法选择：**

| 方法 | 适合 | 成本 | 何时用 |
|------|------|------|--------|
| SFT | 指令跟随、格式遵从 | 低 | **先做**，基座→可用模型 |
| DPO | 偏好对齐，无需奖励模型 | 中 | SFT 之后，偏好微调 |
| RLHF | 复杂偏好、强对齐 | 高 | 需要强对齐时，需训练奖励模型 |

> ⚠️ 默认路径：**SFT 先行 → DPO 进阶 → RLHF 只在需要强对齐时用**。不要一上来就 RLHF，SFT 没做好后面全歪。

**对齐五步：**
1. **选对齐方法** — SFT/DPO/RLHF 按需选择
2. **构建对齐数据** — 高质量指令-响应对 + 偏好数据(chosen/rejected) + 边界 case
3. **执行对齐** — 记录可复现，监控能力回退和拒答泛滥
4. **安全数据与红队** — 有害请求拒答 + 越狱/绕过尝试 + 有害输出检测
5. **双维度评测** — 交 @evaluation，必须能力+安全双达标

**交接标准：** 微调完成 → 交 @evaluation 评测

### Phase 5：评测闸门（核心反馈点①）

| 步骤 | 调用 | 做什么 | 产出 | 对应技能 |
|------|------|--------|------|----------|
| 5.1 | `@evaluation` | 构建评测基准、跑评测、出 Go/No-Go 报告 | 评测报告 + 缺陷清单 | `eval-benchmark-build` |

**示例对话：**

```
@evaluation 给这个模型版本出评测报告，做能力+安全双维度评测
```

**评测三层结构：**

| 层 | 内容 | 用途 |
|----|------|------|
| 自动基准 | 客观指标 + LLM-as-judge | 快速可重复，主评测 |
| 人工抽检 | 真实场景 case | 抓自动指标漏掉的 |
| 红队 | 对抗/越狱/有害 | 安全评测 |

**Go/No-Go 判定：**

```
┌─ @evaluation 判定 ─────────────────────┐
│ Go  → 进入部署流程（Phase 6）            │
│ No-Go → 标注缺陷 → 回流 Phase 3 重训    │
│         （≤2轮，第3轮上报你决策）         │
└────────────────────────────────────────┘
```

**No-Go 重训逻辑（反馈闭环）：**

```
@evaluation: No-Go + 缺陷分析
  → @data-strategy: 补数据（缺什么补什么）
  → @ml-trainer: 加训/换方案
  → @ml-alignment: 重新对齐
  → @evaluation: 再测
  → 仍 No-Go → 再一轮（最多2轮）
  → 第3轮仍不达标 → 上报用户：是数据问题/方案问题/方向问题？
```

> ⚠️ 没有对照基线的评测没意义。"绝对分 90 分"不如"比基线高 5 分"。评测集必须独立于训练数据。

### Phase 6：推理优化 + 封 API + 部署

| 步骤 | 调用 | 做什么 | 产出 | 对应技能 |
|------|------|--------|------|----------|
| 6.1 | `@ml-serving` | 推理引擎选型、量化、延迟/成本优化 | 推理服务 + 对照 benchmark | `ml-inference-optimize` |
| 6.2 | `@backend` | 设计 API（接口+鉴权+配额+计费+错误处理） | API 规范 | `model-api-design` |
| 6.3 | `@mlops` | 数据→训练→评测→上线串成自动化流水线 | 端到端流水线 | `mlops-pipeline` |
| 6.4 | `@infra` | CI/CD 一键部署 + 监控告警 + 回滚方案 | 部署上线 | `infra-cicd-deploy` |

**示例对话：**

```
@ml-serving 评测通过了，做推理优化和部署
@backend 封装模型推理 API，带鉴权和配额
@mlops 把整个流程串成流水线
@infra 一键部署到线上，配监控告警
```

**推理优化关键决策：**

| 引擎 | 特点 | 何时选 |
|------|------|--------|
| vLLM | 高吞吐、PagedAttention | **默认**，大多数场景 |
| TGI | 易用、HF 生态 | HuggingFace 生态 |
| TensorRT-LLM | 极致延迟、Nvidia 优化 | 延迟极致场景 |

| 量化方案 | 精度损失 | 成本节省 |
|----------|----------|----------|
| FP16 | 基准 | 基准 |
| INT8 | 小 | 中 |
| INT4/AWQ/GPTQ | 中 | 大 |

> ⚠️ 量化必须前后对照评测，砍精度要有评测依据。推理成本 ≤ 商业定价 40%（毛利保护）。

**API 设计五要素：**
1. 接口规范（端点 + 请求/响应 schema + 流式）
2. 鉴权配额（API Key + 限流 429 + Retry-After）
3. 计费计量（调用日志支撑计费和分析）
4. 错误处理与边界
5. 文档契约（转 @devrel）

**交接标准：** API 能稳定调用 → @mlops 确认可自动化 → @infra 确认部署通道畅通

### Phase 7：线上 QA 验证 + 反馈闭环（核心反馈点②）

| 步骤 | 调用 | 做什么 | 产出 | 对应技能 |
|------|------|--------|------|----------|
| 7.1 | `@evaluation` | AB 测试 + 灰度发布 + 回滚验证 | AB 报告 + 上线/回滚决策 | `eval-ab-rollout` |
| 7.2 | 不通过 → 回流重训 | 回到 Phase 3 | 循环直到通过 | — |

**示例对话：**

```
@evaluation 设计 AB 测试，灰度验证新模型 API，不通过就回滚重训
```

**灰度策略：**

| 阶段 | 流量 | 观察 |
|------|------|------|
| Canary | 1-5% | 有无崩溃/有害/严重降级 |
| 灰度 | 10-30% | 核心指标对照 |
| 全量 | 100% | 确认无回退 |

**AB 决策逻辑：**

| 结果 | 动作 |
|------|------|
| 主指标升 + 护栏稳 | 全量上线 |
| 主指标平 | 看是否值得换（成本/可维护性） |
| 护栏恶化/事故 | **立即回滚** + 写事故复盘 + 重训 |

**线上事故回滚流程：**

```
@evaluation 发现事故 → @ml-serving 灰度切回旧版 → @infra 确认回滚生效
                   → @ml-alignment + @ml-trainer 定位根因
全程 < 1 小时止血，根因修复进下个迭代
```

## 反馈闭环总结

模型工厂设计了 **两层闭环**，确保不达标不放过：

| 闭环 | 闸门 | 谁判定 | 不通过怎么办 | 最大轮数 |
|------|------|--------|-------------|---------|
| ① 评测闸门 | @evaluation 离线评测 | @evaluation | No-Go → 补数据+重训 | 2 轮（第 3 轮上报用户） |
| ② 线上验证 | @evaluation AB 测试 | @evaluation | 回滚+缺陷分析+重训 | 同上 |

**闭环详细流程：**

```
离线评测 No-Go ─→ 回流数据 ─→ 重训 ─→ 再评测 ─→ 仍 No-Go ─→ 再一轮
                                                              │
                                              第3轮 ─→ 上报用户决策
                                                         │
                                              ┌──────────┤
                                              ▼          ▼
                                           切方向      补数据/换方案
                                           (@ceo)     (@data-strategy/@ml-trainer)

线上 AB 护栏恶化 ─→ 立即回滚 ─→ 定位根因 ─→ 修复/重训 ─→ 重新灰度 ─→ 直到通过
```

**人工确认点：**

| 节点 | 触发条件 | 谁确认 | 为什么 |
|------|---------|--------|--------|
| CEO PRD 走查 | PO 出 PRD 后 | @ceo | 方向错 = 全白干 |
| 评测 No-Go 第 3 轮 | ML 连续 3 轮不达标 | 用户 | 需判断是数据/方案/方向问题 |
| 风控红线 | 涉及安全/合规 | 用户 | 模型安全不可自动放行 |

## 最快路径：一键级联

如果你不想逐步手动调用，模型工厂支持 **级联协议**——只需要和入口角色说"走完流程"，Agent 会自动沿 DAG 往下走：

```
@ceo 我要做法律合同智能审查，走完流程到上线
```

级联自动执行：

```
@ceo → @po 出 PRD → CEO 走查 PRD
  → 通过 → @data-strategy → @data-engineer
    → @ml-trainer → @ml-alignment → @evaluation (Go/No-Go)
      → Go: @ml-serving → @backend → @mlops → @infra → @evaluation(AB)
      → No-Go: 回流重训（自动，≤2轮）
```

**级联触发关键词：** "走完流程""全流程""出模型能力""一键交付"

**不触发级联：** 单一动作（"训个模型""写个 API"）只执行那一步。

## 质疑协议速查

每个关键环节都有自动质疑，防止方向跑偏：

| 产出 | 被质疑者 | 质疑者 | 质疑内容 |
|------|---------|--------|----------|
| PRD | @po | @data-strategy | 数据可得性？训练/评测数据从哪来？ |
| PRD | @po | @evaluation | 评估指标可量化？成功标准能不能测？ |
| 数据标准 | @data-strategy | @ml-trainer | 数据量/质量够不够训练？ |
| 模型 | @ml-trainer | @evaluation | 评测是否充分？ |
| 模型 | @ml-trainer | @ml-serving | 推理可行性/成本？ |
| 重大决策 | @ceo | @evaluation | 有没有数据和评测支撑？ |

## 紧急流程

### 线上模型事故（输出有害/能力回退/服务宕机）

```
@evaluation/监控 发现事故 → @ml-serving 立即灰度回滚上一版本 → @infra 确认回滚生效
                          → @ml-alignment + @ml-trainer 定位根因（数据污染/对齐退化）
全程 < 1 小时止血，根因修复进下个迭代
```

### 模型能力不达标

```
@evaluation 报告评测 No-Go → @data-strategy 补数据 → @ml-trainer 加训/换方案
                            → @po 判断是数据问题还是方向问题 → 若方向错 → @ceo 切方向
不靠"再调调 prompt"硬撑，数据不够就补数据
```

### 能力机会窗口（外部出现新基座/新范式）

```
外部出现更强基座 → @ceo 判断是否值得切 → @ml-trainer 评估迁移成本 → @evaluation 评估效果增益
                → 如果 1 周能切换并出可用能力 → 立刻启动快速出模型能力流程
```

## 快速出模型能力流程（1 周）

> 大模型公司铁律：没有可调用的模型能力，就没有对话。

```
Day 1-2: @po 定场景 + @data-strategy 定数据来源 + 确认可行性
         输入：市场假设 + 算力预算
         输出：1 页 PRD + 数据策略 + 技术方案（自训/微调/基座）

Day 3-4: @ml-trainer 训练/微调 + @ml-alignment 对齐 + @evaluation 跑评测
         输入：数据集 + 技术方案 + 评测基准
         输出：达标模型版本 + 评测报告（通过质疑闸门）
```
