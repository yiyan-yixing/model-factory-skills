# 架构决策记录

## 当前架构

- **阶段**：0->1 创业期
- **模式**：知识蒸馏（教师模型 -> 蒸馏数据 -> 学生模型微调 -> 部署）
- **核心约束**：Mac 48G 统一内存，一人公司，开源基座微调

## 模型蒸馏流水线架构

```
教师模型层        ──→    蒸馏数据层      ──→    学生模型层     ──→    部署层
7B 本地 / API      生成+校验         Qwen2.5 0.5B-3B     mlx 量化推理
(Qwen2.5-7B-4bit)  质量过滤          QLoRA SFT/DPO       FastAPI 服务
                    人工抽检          评估 Go/No-Go        本地 Mac 部署
@data-strategy      @data-engineer    @ml-trainer         @ml-serving
                    @data-strategy    @ml-alignment       @backend
                                      @evaluation         @infra
```

- **教师模型层**：7B 本地 4-bit 推理 + 更强模型 API 调用，生成垂直场景蒸馏数据
- **蒸馏数据层**：教师模型输出 -> 质量过滤 -> 人工抽检 -> 版本化数据集
- **学生模型层**：开源基座 QLoRA 微调 -> SFT 对齐 -> DPO（如可行） -> 评测闸门
- **部署层**：mlx 量化推理 -> FastAPI HTTP 服务 -> 本地 Mac 部署

## ADR（Architecture Decision Records）

格式：`ADR-<编号> | <标题> | <状态>`

### ADR-001: mlx-lm 为首选训练框架

- **状态**：已采纳
- **背景**：Mac 48G Apple Silicon 训练环境需要原生优化的框架
- **决策**：mlx-lm 为首选，PyTorch+MPS 为备选
- **理由**：mlx 针对统一内存架构优化，支持 Qwen2 模型族的 LoRA/QLoRA，社区活跃
- **后果**：训练脚本需从 PyTorch 迁移到 mlx；DPO 暂不支持需备选方案

### ADR-002: 知识蒸馏而非盲微调

- **状态**：已采纳
- **背景**：D6 确定开源基座微调起步，但盲目微调效果不可控
- **决策**：用教师模型定向生成蒸馏数据，而非直接用现有数据微调
- **理由**：蒸馏数据比通用 SFT 数据更精准地迁移特定能力；教师模型的推理能力>学生模型，蒸馏数据天然高于学生可自行学习的水平
- **后果**：增加了教师模型推理环节的复杂度；对教师模型在垂直领域的知识深度有依赖

### ADR-003: 学生模型规模上限 3B

- **状态**：已采纳
- **背景**：Mac 48G 统一内存，QLoRA 训练 3B 约需 12-14GB
- **决策**：学生模型不超过 3B 参数
- **理由**：3B QLoRA 在 48G 上可行且留有余量；7B QLoRA 需极限优化速度极慢
- **后果**：只能做分类/抽取/简单生成任务，不做深度推理任务
