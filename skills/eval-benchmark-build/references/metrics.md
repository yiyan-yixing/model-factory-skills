# 评测指标与局限

> 选错指标 = 优化错方向。**每个指标都有盲区，组合使用 + 人工抽检才能看清真实质量。**

## 按任务类型选指标

### 分类任务

| 指标 | 用途 | 注意 |
|------|------|------|
| Accuracy | 整体对错率 | 类别不平衡时会骗人 |
| Precision / Recall | 关注某类 | 抽取/检测任务必看 |
| F1 | P/R 平衡 | 不平衡数据首选 |

> 类别极不平衡时，Accuracy 95% 可能毫无意义（全猜多数类就有 95%）。

### 抽取 / 问答（QA）

| 指标 | 用途 |
|------|------|
| Exact Match (EM) | 答案完全匹配的比例（严） |
| Token-level F1 | 部分匹配给部分分（松） |
| Soft accuracy | 语义等价算对（LLM 判） |

### 文本生成

| 指标 | 局限（重要） |
|------|-------------|
| BLEU | n-gram 重叠，**和人类感知相关性差**，只适合翻译/摘要等有参考的 |
| ROUGE | 同上，召回侧重 |
| **不要单独用 BLEU/ROUGE 评开放生成** | 会得到"指标高但人看不行"的结果 |

### 代码

| 指标 | 说明 |
|------|------|
| pass@k | k 次采样至少 1 次通过的功能正确率 |
| pass@1 | 最常用（单次生成通过率） |
| functional correctness | 跑测试用例，对就是对 |

> 代码评测要**真跑测试**，不是看像不像。用 HumanEval/MBPP 的执行式评测。

### 数学

| 指标 | 说明 |
|------|------|
| Exact match（最终答案） | 只比对最终数字/表达式 |
| Majority voting | 多次采样取多数（降随机性） |

### 指令跟随 / 格式

| 指标 | 说明 |
|------|------|
| 格式遵从率 | 输出是否符合要求的 schema（JSON/字段） |
| 指令遵循率 | 是否满足约束（长度/语气/必含项） |

### 偏好 / 整体质量

| 指标 | 说明 |
|------|------|
| Win rate | A vs B 胜率（人工或 LLM judge） |
| Elo / Bradley-Terry | 多模型两两比较排序 |

## 通用陷阱（必须知道）

| 陷阱 | 说明 | 对策 |
|------|------|------|
| **指标好 ≠ 体验好** | BLEU 高但读着别扭 | 配人工抽检 |
| **数据污染** | benchmark 数据泄露进训练集 → 虚高 | 用新/私有 benchmark 复核 |
| **单一指标过拟合** | 只优化一个指标，牺牲其他 | 多指标 + 护栏 |
| **绝对分无意义** | "85 分"脱离基准看不懂 | 必须对照基线/竞品 |
| **分布偏差** | 评测集不代表真实用法 | 用真实流量采样补评测集 |

## 指标组合建议

一个能力的评测应该包含：
1. **1 个主指标**（最能反映该能力价值）
2. **2-3 个护栏指标**（不能恶化的维度，如延迟/格式/安全）
3. **人工抽检**（抓指标盲区）
4. **对照基线**（旧版本/开源基座/竞品）

> 没有"一个指标评一切"。组合 + 对照 + 抽检，才接近真实。
