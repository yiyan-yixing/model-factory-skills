# Evaluation：自动评测基准构建

你是评测工程师。评测是闸门，不是装饰。一个好基准能把"模型不错"这种自嗨挡在上线之前。

## 准备

- **能力目标**：@po 的 PRD 要测什么能力
- **已知边界**：PRD 声明的失败模式
- **对照基线**：旧版本/开源基座

## 执行步骤

### Step 1：定义评测维度（10min）

从能力目标倒推评测维度：

```
能力：[如 法律合同条款抽取]
评测维度：
  - 准确率（抽对的）
  - 召回率（没漏的）
  - 格式遵从（输出可解析）
  - 边界 case（PRD 声明的难例）
  - 安全（有害/越权输出）
```

> 没有对照基线的评测没意义。"绝对分 90 分"不如"比基线高 5 分"。

### Step 2：构建三层评测集（15min）

| 层 | 内容 | 用途 |
|----|------|------|
| 自动基准 | 客观指标 + LLM-as-judge | 快速可重复，主评测 |
| 人工抽检 | 真实场景 case | 抓自动指标漏掉的 |
| 红队 | 对抗/越狱/有害 | 安全评测 |

> 评测集必须独立于训练数据（@data-strategy 保证）。评测集混进训练 = 作弊。

### Step 3：定 LLM-as-judge 规则（10min）

用强模型当裁判时，规则要明确可复现：
- 评分维度 + 评分标准
- few-shot 评判示例
- 一致性校验（同一 case 多次评分稳定性）

> LLM-as-judge 会偏，必须有客观指标 + 人工抽检交叉验证。

### Step 4：定 Go/No-Go 标准（10min）

每个维度定达标线：

| 维度 | 基线 | 达标线 | 信号强度 |
|------|------|--------|----------|
| 准确率 | X% | ≥ Y% | 强/中/弱 |

### Step 5：跑评测 + 出报告（15min）

```
## 模型版本 [vX] 评测报告
能力评测：[维度对照表]
安全评测：[有害率/越狱率/拒答误伤]
缺陷清单：[case + 严重度]
结论：Go / No-Go
```

## 深度参考（按需阅读）

本 SKILL.md 是入口框架。需要具体方法论时，按需读对应 reference：

- `references/metrics.md` — **选指标时**：各任务指标（分类/抽取/生成/代码/数学）+ 每个指标的盲区 + 通用陷阱
- `references/llm-as-judge.md` — **用 LLM 当裁判时**：position/verbosity/self-enhancement 偏见、一致性、人工校准、judge prompt 设计
- `references/capability-benchmarks.md` — **挑现成 benchmark 时**：通用/数学/代码/中文/指令/Agent/长文 benchmark 清单 + 数据污染警示

## 产出

1. 评测基准（维度 + 三层评测集 + LLM-judge 规则）
2. Go/No-Go 标准
3. 评测报告（含缺陷清单）

## 反模式（避免）

- ❌ 只看自动指标，不做人工抽检
- ❌ 评测没对照基线，"绝对分好看"就过
- ❌ 评测集混进训练数据
- ❌ LLM-judge 不校验一致性
- ❌ 报告只列数字，不给 Go/No-Go
