# 训练失败诊断手册

> 训练炸了别慌。90% 的不稳定能归到下面几类。先按症状定位，再对症。
> 越早止损越省算力——loss 发散硬撑是最贵的错误。

## 症状 → 原因 → 对策 速查

### 1. Loss 发散 / 飙升（loss spike 后不回来）

| 可能原因 | 怎么验证 | 对策 |
|----------|----------|------|
| LR 太大 | 看 spike 前的梯度范数是否飙升 | 降 LR（降 2-10 倍），加梯度裁剪 |
| batch 太小梯度噪声大 | 算有效 token batch | 增大梯度累积 / 数据并行 |
| 数据里有坏样本（超长/乱码/NaN） | 检查 spike 时刻的数据 batch | 加数据过滤，记录坏 batch |
| 数值不稳定（fp16 溢出） | 看是否有 inf/NaN | 上 bf16，或加 loss scaling |

> **第一反应永远是降 LR + 查那个 batch 的数据。** 别急着换架构。

### 2. Loss 是 NaN

| 可能原因 | 对策 |
|----------|------|
| 数值溢出（fp16） | 换 bf16 / fp32 关键层 |
| 输入含 NaN/Inf | 数据层加 NaN 检查过滤 |
| LR 太大爆炸 | 降 LR，从最后一个稳定 checkpoint 续训 |
| 某层梯度爆炸 | 检查梯度范数，加裁剪 |

> 续训：从 spike/NaN 前的 checkpoint 重启，**跳过**导致问题的那批数据，降 LR。

### 3. Loss 不降 / 降得慢

| 可能原因 | 对策 |
|----------|------|
| LR 太小 | LR range test，找陡降区间 |
| warmup 太长 | 缩短 warmup |
| 数据质量差/重复多 | 洗数据、去重 |
| 学习率 schedule 错误 | 检查 cosine decay 配置 |

### 4. 显存 OOM

| 场景 | 对策 |
|------|------|
| 模型放不下 | 激活重计算（gradient checkpointing）、ZeRO-2/3、FSDP |
| batch 太大 | 降单卡 batch，梯度累积补 |
| 序列太长 | 用 sequence parallelism / 降 seq_len / 分块注意力 |
| 优化器状态爆 | ZeRO-3 分片优化器状态 |

> OOM 优化顺序：**激活重计算（省最多）→ 优化器分片（ZeRO/FSDP）→ 梯度累积**。通信换显存，按需上张量并行（TP，跨机通信贵）。

### 5. 训练正常但下游评测差

| 可能原因 | 对策 |
|----------|------|
| 训练目标 ≠ 下游能力（只看 loss） | 看下游指标，不只看 loss |
| 数据域配比不对 | 调 domain mixing（见 scaling-and-data-mixing） |
| 过拟合（重复数据） | 去重、加多样性 |
| 评测基准本身有问题 | 让 @evaluation 复核基准 |

> **训练 loss 好看不等于模型好。** 必须看下游评测——这是为什么评测是闸门。

## 训练稳定的 checklist（启动前过一遍）

- [ ] LR 做过 range test 或用了同规模经验值
- [ ] 梯度裁剪开了（~1.0）
- [ ] 数值精度用 bf16（或 fp16 + loss scaling）
- [ ] 数据去过重、过滤过坏样本
- [ ] checkpoint 频率够（崩了不丢太多进度）
- [ ] 有上一稳定 checkpoint 可续训
- [ ] 监控在记：loss / 梯度范数 / LR / 显存 / 吞吐

## 省算力的纪律

- **小规模先验证**：超参/数据/代码先用 1/10 算力跑通，再 scale up
- **early stop**：loss 异常立刻停，别等"再跑跑看"
- **可复现**：每个实验记 超参+数据版本+种子，崩了能复现、能对比

> 训练是烧钱的。每次"再跑跑看"都在烧钱，纪律比技巧值钱。
