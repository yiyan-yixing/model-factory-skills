# Scaling Law 与数据配比

> 决定"投多少算力、用多少数据、训多大模型"的判断依据。
> 核心结论：**很多团队训练效率低，是因为数据不够 / 配比不对，不是因为模型不够大。**

## Chinchilla：计算最优配比

经典结论（Hoffmann et al. 2022）：固定计算预算 C 时，最优的 **模型参数 N** 与 **训练 token D** 应近似按 ~20 token/参数 配比。

| 模型参数 | 计算最优 token 数（约） |
|----------|----------------------|
| 1B | ~20B |
| 7B | ~140B |
| 70B | ~1.4T |

> 现实：很多开源模型是 **over-trained**（数据远超 20:1，如用更多数据训小模型换推理便宜）。这是**有意权衡**——多花训练算力换推理成本下降。是否 over-train 取决于"训练算力 vs 终身推理调用量"的权衡。

## 决策：给固定算力，怎么分？

```
你有 X GPU·小时预算 →
  选项 A：训大模型 + 少数据（undertrained，效果差）
  选项 B：训小模型 + Chinchilla 数据（计算最优）
  选项 C：训小模型 + 过量数据（over-trained，推理便宜）← 调用量大时选这个
```

> 15 人公司几乎一定选 B 或 C，不选 A。除非有海量独占数据，否则别训超大模型。

## 数据配比（domain mixing）— 实战最容易被忽视

预训练不是"有什么数据全倒进去"。不同域（网页/代码/数学/书籍/对话）的**配比**显著影响能力分布。

### 配比的经验锚点（聚合自多家报告，需按目标调）

| 数据域 | 典型占比区间 | 作用 |
|--------|-------------|------|
| 网页文本 | 30-60% | 通用语言、广度 |
| 代码 | 5-30% | 推理、代码能力、结构化 |
| 数学/科学 | 5-15% | 推理能力 |
| 书籍/长文 | 5-20% | 长程连贯、知识深度 |
| 对话/指令样式的预训练数据 | 0-10% | 改善对话性 |

> 想要代码能力强 → 代码占比拉高（有的代码模型代码占 30%+）。想推一般推理 → 加数学/科学。

### 配比调法的陷阱

- **配比不是一次定死**：通常分阶段（先大比例通用打底，后期加高质量域）
- **数据质量 > 配比 > 数据量**：低质代码 30% 不如高质代码 10%
- **重复数据有毒**：高重复会降泛化、增记忆，去重是必做项（见数据流水线）
- **不要被某家报告的精确配比绑架**：那是别人的数据分布，你要在自己的评测集上验证

## "够不够训"的快速判断

回答这个问题（@data-strategy 也会问）：

1. **数据量**：达到 Chinchilla ~20:1 了吗？达不到 → 考虑 over-train 小模型 或 补数据
2. **数据质量**：去过重、去毒、质量过滤了吗？没有 → 先洗
3. **域覆盖**：目标能力的域数据够吗？不够 → 能力会弱
4. **多样性**：分布单一吗？单一 → 容易过拟、泛化差

> 数据不够别硬训大模型。先补数据或缩模型规模，是更省钱的路径。
