# 流水线架构设计

> MLOps 的核心不是"用哪个工具"，是把 **数据→训练→评测→上线** 串成可重放、可观测、带闸门的自动化流水线。
> 这一页讲流水线怎么分 stage、怎么编排、怎么设计才不脆。

## 标准流水线 stage

```
[数据准备] → [训练] → [评测] → [候选判定] → [打包] → [灰度上线] → [监控]
     ↑          ↑        ↑                                    │
   可重放     可复现    闸门(Go/No-Go)                         │
     └──────────┴────────┴──── 不达标告警 + 回流数据 ←────────┘ (监控退化触发重训)
```

每个 stage 的契约：

| Stage | 输入 | 输出 | 闸门 |
|-------|------|------|------|
| 数据准备 | 原始数据 | 版本化干净数据集 | 质量校验通过 |
| 训练 | 数据版本 + 超参 | checkpoint + 训练日志 | loss 正常收敛 |
| 评测 | checkpoint | 评测报告 + Go/No-Go | 指标达标 |
| 候选判定 | 评测通过 | 候选模型版本 | 评测说 Go |
| 打包 | 候选模型 | 推理镜像/服务包 | 能加载能调 |
| 灰度上线 | 服务包 | 线上版本 | AB 指标不恶化 |
| 监控 | 线上指标 | 健康/退化信号 | 退化触发重训 |

> **评测是闸门**：不达标的模型自动拦截，不能进"打包/上线"。这是流水线防自嗨的关键。

## 编排工具选型

| 工具 | 特点 | 适合 |
|------|------|------|
| **Airflow** | 成熟、生态广、定时 DAG 强 | 通用定时流水线 |
| **Kubeflow Pipelines** | K8s 原生、ML 友好 | 已上 K8s 的团队 |
| **Prefect / Dagster** | 现代、数据/资产导向 | 数据资产驱动 |
| **Argo Workflows** | K8s 原生、轻量 | 容器化任务编排 |
| **自研 DAG** | 完全可控 | 特殊需求（但维护成本高） |

> 选型原则：**先用最熟/最简单的跑通**，别为了用 Kubeflow 而上 K8s。15 人团队 Airflow 或 Prefect 通常够。

## 触发方式

| 触发 | 场景 |
|------|------|
| 手动 | 实验期、调试 |
| 定时 | 周期性再训（如每周） |
| 数据驱动 | 新数据累积到阈值自动触发（持续训练） |
| 退化驱动 | 监控发现线上指标退化触发重训 |
| CI | 代码/配置变更触发小规模验证 |

## 让流水线不脆的设计原则

- **幂等**：同一输入重跑得到同一结果（依赖版本固定：数据/代码/环境）
- **可重放**：能从任意 stage 重跑（数据版本 + 超参 + 种子都记着）
- **可观测**：每步状态、日志、产物可查；失败能定位到具体 stage
- **有闸门**：评测自动拦截劣质模型，不靠人盯
- **可回滚**：上线出问题能切回上一版本（对接 @infra）
- **失败不静默**：任何 stage 失败都告警，不"跑完以为成功"

## 流水线成熟度（你在哪一级）

| 级级 | 特征 |
|------|------|
| L0 | 全手动，脚本各自为政 |
| L1 | 部分自动化，但靠人搬运 |
| L2 | 端到端流水线，但无闸门 |
| L3 | 流水线 + 自动评测闸门 + 可回滚 ← **目标** |
| L4 | 持续训练（数据/退化自动触发）+ 全自动 |

> 15 人公司目标 **L3**。L4 需要成熟的监控和评测自动化，逐步演进。
