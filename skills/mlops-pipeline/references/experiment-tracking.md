# 实验追踪

> 没有实验追踪 = 训练完说不清"为什么这个好""怎么复现""哪个超参起作用"。
> 实验追踪让每次训练**可记录、可对比、可搜索、可复现**。

## 主流工具

| 工具 | 强项 |
|------|------|
| **W&B（Weights & Biases） | UI 强、协作好、生态广，业界主流 |
| **MLflow** | 开源、轻量、可自托管、框架中立 |
| **Aim** | 开源、实验对比快、轻量 |
| **TensorBoard** | 基础、训练曲线，不够管理多实验 |

> 通用首选 W&B 或 MLflow（自托管偏好选 MLflow）。核心是**全链路记录**，工具其次。

## 一次实验必须记录什么（全链路）

| 类别 | 记录内容 |
|------|----------|
| **代码** | 代码版本（git commit）、关键配置文件 |
| **数据** | 数据集版本 + hash + 来源 |
| **超参** | 全部超参（LR/batch/调度/种子/梯度裁剪…） |
| **环境** | 框架版本、硬件、分布式配置 |
| **指标** | 训练 loss、验证指标、下游评测分（随步数曲线） |
| **产物** | checkpoint 路径 + 版本 |
| **状态** | 运行状态、起止时间、谁跑的 |

> 记不全 = 实验不可复现。"我大概记得改了 LR"不算追踪。

## 让实验可对比、可搜索

- **命名规范**：实验名含关键变量（如 `exp-llama7B-lr3e4-dataV3`），一眼能看出在测什么
- **标签/分组**：按目标/数据版本/方法打 tag，方便筛选对比
- **基线固定**：每次迭代都对照同一基线，看相对提升
- **记录假设**：这个实验验证什么假设，结果是否成立——不只记数字

## 可复现性检查

> 可复现 = 给同样的代码/数据/超参/种子，能跑出同样结果。

- 随机种子固定（数据 shuffle、初始化、dropout）
- 数据版本固定（hash 对得上）
- 依赖固定（环境锁定）
- 分布式可能有微小差异（接受或记录）

> 实验追踪的最高标准：**别人拿你的记录能复现你的结果**。做不到就不是真追踪。

## 实验管理的反模式

| 反模式 | 后果 |
|--------|------|
| 只存 checkpoint 不存超参/数据版本 | 无法复现、无法对比 |
| 实验名 `exp1`/`exp2`/`test_new` | 几周后没人记得测了什么 |
| 不记假设和结论 | 数字一堆，没人知道说明了什么 |
| 训练 loss 好看就停 | 没看下游评测 |
| 基线每次变 | 相对提升无意义 |

## 与其他角色的衔接

- @ml-trainer 训练 → 你把实验全链路记入追踪系统
- @evaluation 评测 → 评测结果回写到同一实验记录（训练+评测在一个视图）
- @ml-serving 上线 → 上线版本关联到具体实验（线上版本可追溯到训练实验）

> 追踪系统是全闭环的"记忆中枢"：训练→评测→上线，一个版本号串起来。
