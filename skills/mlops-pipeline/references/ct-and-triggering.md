# 持续训练与自动触发

> 模型不是"训一次就完"。数据在长、分布会漂、线上会退化——**持续训练（CT）让模型持续保鲜**。
> 但 CT 的难点不是"再训一次"，是"什么时候自动训、训完怎么自动判断能不能上"。

## 什么时候触发重训

| 触发信号 | 说明 | 例子 |
|----------|------|------|
| **数据累积** | 新数据攒到阈值 | 新增 N 万条垂直数据 |
| **定时** | 周期性再训 | 每两周一次例行再训 |
| **线上退化** | 监控发现指标下滑 | 线上准确率/满意度下降 |
| **分布漂移** | 输入分布变了 | 用户问的东西变了 |
| **能力补强** | 要加新能力 | 新增一个评测维度 |

> 触发要**有据**：不是"感觉该训了"，是数据量/退化信号/分布漂移的量化阈值。

## 自动评测闸门（CT 的核心）

CT 必须配自动评测，否则就是"自动生产垃圾"：

```
触发重训 → 自动训练 → 自动评测
                        ├─ 达标 → 进候选池 → 灰度上线（配合 @evaluation AB）
                        └─ 不达标 → 告警 + 分析（数据不够？超参不对？回归了？）
```

- **达标线自动判定**：指标过线才进候选，不靠人放行
- **护栏自动检查**：延迟/成本/安全不能恶化
- **对照基线**：必须比当前线上版本好（或至少不差 + 其他维度提升）才上

> 没有自动闸门的 CT = 自动上线没验证的模型 = 定时炸弹。

## 候选模型管理

- 每次重训产出**候选模型**，进候选池（不直接上线）
- 候选模型带完整元信息（数据版本/超参/评测分/触发原因）
- 灰度时从候选池取，AB 通过才转正
- 淘汰的候选归档（保留追踪，不直接删）

## 退化监控（驱动 CT 的输入）

CT 要有效，**线上退化监控**必须先做好：

| 监控 | 说明 |
|------|------|
| 质量退化 | 线上输出质量评分下降（用户反馈/抽检/LLM judge） |
| 分布漂移 | 输入分布偏离训练分布 |
| 投诉/负面反馈 | 用户负面信号上升 |
| 特定能力退化 | 某类 query 效果变差 |

> 没有退化监控 = 不知道该重训。CT 的前提是可观测。

## CT 的成熟度（别一上来就全自动）

| 级别 | 特征 |
|------|------|
| 手动重训 | 人决定训、人判断能不能上 |
| 半自动 | 训练自动化，但评测/上线靠人 |
| 闸门自动 | 评测自动判定 Go/No-Go |
| **持续训练** | 触发→训练→评测→上线全自动，人只监督 ← 高级目标 |

> 15 人公司先做到"闸门自动"（L3），CT 全自动（L4）是演进终点，别强上。

## CT 的风险与对策

| 风险 | 对策 |
|------|------|
| 自动上线劣质模型 | 闸门 + 灰度 + 回滚（必须有） |
| 数据累积触发过频，烧算力 | 设触发阈值 + 预算上限 |
| 持续训练导致回归（新数据污染） | 每次对照旧版评测，回归即拦 |
| 全自动出问题没人发现 | 监控 + 告警 + 人在回路的关键节点 |

> CT 不是"无人值守"。是"自动执行 + 人在关键节点监督 + 出问题能停"。
