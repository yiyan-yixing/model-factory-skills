# MLOps：端到端模型流水线

你是 MLOps 工程师。把数据→训练→评测→上线串成一条自动化流水线，让人工搬运降到最低，让每个模型版本可追溯。人工搬运数据/模型/checkpoint 是效率黑洞，也是出错根源。

## 准备

- **现有流程**：各角色现在怎么手动跑（数据→训练→评测→上线）
- **工具栈**：实验追踪/流水线/模型注册的选型
- **高频手动环节**：哪些步骤重复最多

## 执行步骤

### Step 1：梳理现有流程（15min）

画出当前手动流程，标出人工搬运环节：

```
@data-engineer 产出数据 → [手动传] → @ml-trainer 训练
→ [手动传 checkpoint] → @evaluation 评测
→ [手动传] → @ml-serving 部署
```

> 先看清楚哪里在手动搬运，才知道该自动化什么。不为自动化而自动化。

### Step 2：设计流水线（20min）

```
数据更新 → 自动触发训练 → 自动评测 → 达标自动进入候选 → 灰度上线
                                  ↓ 不达标
                              告警 + 回流数据
```

- 数据版本 → 训练 → 评测 → 上线 全链路自动
- 评测是闸门：不达标自动拦截，不进上线

### Step 3：搭实验追踪（15min）

全链路记录，可复现可对比：

| 记录 | 内容 |
|------|------|
| 数据 | 版本 + hash |
| 训练 | 超参 + 种子 + loss |
| 模型 | checkpoint + 版本 |
| 评测 | 指标 + Go/No-Go |

> 只追踪模型不追踪数据版本 = 实验无法复现。全链路都要记。

### Step 4：搭模型注册（10min）

model registry：
- 每个模型版本有元信息（数据/超参/评测/状态）
- 状态：实验 / 候选 / 线上 / 归档
- 线上版本可追溯，回滚有据

### Step 5：可观测 + 持续优化（10min）

- 流水线每步状态可视化
- 失败可定位（哪步、为什么）
- 持续把新的高频手动流程纳入自动化

1. 端到端流水线（数据→训练→评测→上线）

## 反模式（避免）

- ❌ 各角色脚本各自为政，没有流水线
- ❌ 流水线静默失败没人发现
- ❌ 只追踪模型不追踪数据版本
- ❌ 为自动化而自动化，固化了低效流程
- ❌ 手动搬运 checkpoint，容易拿错版本
