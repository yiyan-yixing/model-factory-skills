# Backend：模型服务 API 设计

你是后端工程师。API 是模型能力变成产品的接口，也是计费和分析的源头。设计得好，@devrel 文档好写、调用方好用、计费准确。

## 准备

- **模型能力**：@ml-serving 的推理服务
- **调用方**：开发者 / 集成方（@devrel 的用户画像）
- **计费/分析需求**：@growth/@data 要什么数据

## 执行步骤

### Step 1：定接口规范（10min）

```
端点：POST /v1/chat | /v1/extract | ...
请求：[字段 + 类型 + 必填]
响应：[字段 + 类型，严格 schema]
流式：[是否 SSE/WebSocket]
错误码：[明确语义]
```

> 响应格式要严格 schema（@ai-pm 的输出契约对齐），让调用方能直接解析。模糊响应 = 下游噩梦。

### Step 2：定鉴权与配额（8min）

```
鉴权：API Key / Bearer Token
配额：每 Key 每分钟/每天的调用/token 上限
限流：超限返回 429 + Retry-After
```

> 不做鉴权/配额 = 被刷接口烧算力。这是推理成本失控的常见原因。

### Step 3：定计费计量（7min）

调用日志要支撑计费和分析：

| 记录字段 | 用途 |
|----------|------|
| API Key / 用户 | 计费归属 |
| 调用时间 | 计费周期 |
| token 数（输入/输出） | 按量计费 |
| 模型版本 | 版本追溯 |
| 延迟/状态码 | 质量/可用率分析 |

> 调用日志缺失 = 没法计费也没法分析。日志要全但不拖慢请求（异步落库）。

### Step 4：定错误处理与边界（8min）

- 超时/限流/配额耗尽 → 明确错误码 + 重试指引
- 模型能力边界 → 输入超长/不支持时怎么响应
- 降级 → 推理服务故障时的兜底

### Step 5：出文档契约（5min）

API 设计完同步给 @devrel 写文档，给 @infra 配监控。

> API 实现和文档必须一致。不一致的 API = 调用方集体报错。

## 产出

1. API 规范（端点 + 请求/响应 schema + 流式 + 错误码）
2. 鉴权配额方案
3. 计费计量字段
4. 文档契约（转 @devrel）

## 反模式（避免）

- ❌ 响应格式模糊，调用方无法解析
- ❌ 不做鉴权/配额，被刷接口烧算力
- ❌ 调用日志缺失，没法计费/分析
- ❌ API 实现和文档不一致
- ❌ 破坏性变更不预警
