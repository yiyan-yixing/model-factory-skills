# 推理成本核算与 GPU 选型

> 推理成本是大模型公司的生命线。**模型训得好但推理亏本 = 做慈善。**
> 这一页给你算清单 token 成本、选对 GPU、把利用率榨干。

## 单 token 成本公式

核心公式：

```
单 token 成本 ≈ GPU 秒单价 × (生成每 token 耗时) / (并发请求分摊)
```

或等价地，用吞吐反推：

```
单 token 成本 = GPU 时薪 / (每小时产出 token 数)
每小时产出 token 数 = 吞吐(tokens/s/卡) × 卡数 × 3600
```

> 关键变量：**吞吐**（tokens/s）。吞吐翻倍 = 单 token 成本减半。这就是为什么 continuous batching / 量化 / prefix cache 都在"省钱"。

## 成本速算示例（教学用，数字会变）

假设：
- GPU 租用 ¥X/卡/小时
- 单卡吞吐 Y tokens/s（你的实测值）
- 则：单千 token 成本 ≈ X / (Y × 3.6) 元

> 把 X、Y 换成你的**实测**值。别用厂商理论吞吐，实测通常打折扣。

## GPU 选型要素

| 要素 | 为什么重要 |
|------|-----------|
| **显存** | 模型权重 + KV cache 能不能装下，决定能用几张卡、多大 batch |
| **显存带宽** | 推理是**访存密集**（decode 阶段），带宽往往比算力更决定速度 |
| **算力** | prefill 阶段、大 batch 时重要 |
| **单价** | 性价比 = (吞吐/价格)，不是单看任一项 |

> **大模型推理（尤其 decode）瓶颈常在显存带宽，不是算力。** 选卡别只看 TFLOPS，看带宽和显存。

## 常见 GPU 特征（定性，具体看最新规格）

- H 系列：FP8 支持、高带宽、贵 → 吞吐/质量优先的大流量服务
- A 系列：成熟稳定、性价比好 → 通用主力
- 消费级卡：显存小但便宜 → 小模型/边缘，注意显存限制和 EULA

## 把利用率/成本榨干的杠杆（按 ROI 排序）

1. **continuous batching**（不开 = 浪费 90% 吞吐）— ROI 最高
2. **量化**（INT8/FP8 几乎无损省一半显存，多塞并发）
3. **prefix caching**（重复前缀复用，Agent/多轮场景）
4. **右尺寸 GPU 数**（别为峰值长开空闲卡，用自动扩缩容）
5. **spot/预留实例混合**（稳定部分预留，弹性部分 spot）

## 成本可控的纪律

- **推理成本 ≤ 商业定价的 40%**（毛利保护线，经验值，按你毛利目标调）
- **持续监控单 token 成本趋势**——模型变重/流量结构变，成本会偷偷涨
- **峰值不是常态**——按 P50/P80 容量规划，峰值靠弹性扩容，别按峰值常开

## 成本核算 checklist（上线前）

- [ ] 实测了单卡吞吐（tokens/s），不是理论值
- [ ] 算出了单 token 成本
- [ ] 对照了商业定价，毛利 ≥ 目标
- [ ] 知道瓶颈在哪（算力/显存/带宽）
- [ ] 定了自动扩缩容策略（不按峰值常开）
- [ ] 有成本监控告警（单 token 成本异常飙升能发现）
