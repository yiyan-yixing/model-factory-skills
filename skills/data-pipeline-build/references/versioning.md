# 数据版本管理

> 没有数据版本 = 训练结果无法复现、无法对比、无法回溯"这个模型用了哪批数据"。
> 数据要像代码一样**可版本、可追溯、可 diff**。

## 为什么数据版本化是刚需

| 场景 | 没有版本化 | 有版本化 |
|------|-----------|----------|
| 模型效果变了 | "不知道数据改了啥" | diff 数据版本，定位变化 |
| 复现实验 | 不可能 | 取同一数据版本重跑 |
| 线上事故 | "这模型用的什么数据？" | 查模型→数据版本→来源 |
| 对比实验 | 数据都不一样怎么比 | 固定数据版本对比 |

> 数据版本和模型版本必须**一一对应**。模型出问题，能立刻查到它吃了哪批数据。

## 版本化要记录什么

每个数据集版本至少记录：

```
dataset: <名称>-v<N>
来源: [来源清单版本]
清洗规则: [清洗配置版本]
条数: N
去重率/质量指标: ...
hash: <整体 hash>
创建: <日期>
衍生自: <上一版本（如果有）>
```

> hash 是关键——同一版本 hash 必须一致，才能确认"真的是同一批数据"。

## 工具选择

| 工具 | 特点 |
|------|------|
| **DVC** | Git 友好、数据版本主流选择 |
| **对象存储 + 元数据** | 简单，路径/manifest 记版本 |
| **LakeFS** | 数据像代码一样分支/合并 |
| **数据集卡片 + hash** | 轻量，自托管够用 |

> 15 人公司：**DVC 或 对象存储+manifest** 通常够，别上重系统。

## 数据版本 ↔ 模型版本 ↔ 实验

闭环的关键——三者绑定：

```
模型版本 M3
  └─ 训练自实验 exp-007
       └─ 用数据 dataset-v5 (hash: ...)
            └─ 清洗自原始 raw-v3
```

- @ml-trainer 训练时记录用了哪个数据版本
- 实验追踪系统（见 experiment-tracking）把数据版本作为实验输入
- @ml-serving 上线时，线上版本关联到训练实验→数据版本

> **线上任何模型都能一路追溯到它用的原始数据**。这是数据合规、事故排查、效果归因的基础。

## 数据 diff

版本化的进阶用法——**对比两个数据版本的差异**：
- 新增了多少 / 删除了多少 / 修改了多少
- 分布有没有变
- 哪些域变了

> 模型效果突变时，数据 diff 是第一现场。常能直接发现"哦这批数据混入了 X"。

## 版本化的反模式

| 反模式 | 后果 |
|--------|------|
| 数据集没版本号 | 无法复现/对比/回溯 |
| 只存最终数据不存来源 | 不知道怎么来的，无法审计 |
| hash 不固定 | "同名数据"其实是两批 |
| 清洗规则不版本化 | 数据版本含义不清 |
| 模型不关联数据版本 | 线上模型查不到用了什么数据 |

## 与其他角色的衔接

- @data-strategy 定数据需求 → @data-engineer 产出**版本化数据集**
- @ml-trainer 训练必须**记录数据版本**（实验输入）
- @evaluation 评测数据**独立版本化**（且不与训练数据混）
- 线上模型 → 数据版本可追溯（合规/审计/归因）

> 数据版本化是**全闭环可追溯的地基**。没有它，"可复现""可归因""可审计"全是空话。
