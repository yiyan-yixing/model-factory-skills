# 偏好数据构建

> 对齐的天花板由偏好数据质量决定。**垃圾偏好进，垃圾模型出。**
> 这一页讲数据怎么造、怎么验、怎么不踩坑。

## 数据形态

### SFT 数据

```
{ "instruction": "...", "input": "...", "output": "高质量标准答案" }
```
- 一条 = 一个 (指令, 期望输出)
- 质量 > 数量；几千~几万条高质量就能显著改善指令跟随

### 偏好对（DPO/RLHF 用）

```
{ "prompt": "...", "chosen": "更好的回答", "rejected": "较差的回答" }
```
- chosen 和 rejected 必须**针对同一个 prompt**
- 差距要**清晰**（模糊的偏好模型学不到信号）

### 二值反馈（KTO 用）

```
{ "prompt": "...", "response": "...", "label": "good/bad" }
```
- 来自"赞/踩"、点赞/点踩的真实反馈

## 偏好数据从哪来

| 来源 | 质量 | 成本 | 注意 |
|------|------|------|------|
| 人工标注（专家写 chosen/rejected） | 最高 | 高、慢 | 标注一致性要校验 |
| 用强模型生成偏好（distillation） | 中高 | 低、快 | 版权/许可；可能引入强模型偏见 |
| 真实用户反馈（赞/踩） | 中（但有真实信号） | 低 | 偏稀疏、有偏（活跃用户） |
| 规则/启发式（如长度、格式） | 低 | 极低 | 仅作弱信号，单独用会学到表面特征 |

> 主流：**强模型 distillation 起步 + 人工精标关键域 + 用户反馈持续补**。

## 质量门槛（收数据前定好）

- **标注一致性**：多人标同一批，一致率 < 70% → 说明任务定义不清，先改标注规范
- **chosen-rejected 差距清晰度**：人工抽检，模糊对（差不多好）要剔除
- **覆盖度**：覆盖目标能力的核心场景 + 边界 case + 安全 case
- **无污染**：偏好数据**不能混进评测集**（否则评测作弊）

## 常见污染与陷阱

| 陷阱 | 后果 | 对策 |
|------|------|------|
| 偏好数据混入评测集 | 评测虚高、自欺 | 数据集严格隔离 + hash 去重 |
| chosen 只是"更长" | 模型学会啰嗦 | 控制长度变量，加长度惩罚评测 |
| rejected 太差（一好一烂） | 学不到细粒度偏好 | 让 rejected 是"还行但不够好" |
| 全是强模型生成的偏好 | 继承强模型偏见/价值观 | 混入人工标注校正 |
| 偏好数据分布与真实用法不符 | 对齐了错误的偏好 | 用真实用户反馈校准分布 |
| 偏好标注者偷懒/不一致 | 数据噪声大 | 一致性校验 + 黄金题抽检 |

## 规模经验（聚合值，需校准）

| 阶段 | 偏好对数量级 | 说明 |
|------|-------------|------|
| 起步验证 | 数千~数万 | 验证 pipeline、看是否有信号 |
| 可用 | 数万~数十万 | 多数垂直场景够用 |
| 通用强对齐 | 百万级 | 通用大模型级别 |

> 垂直场景**几万条高质量偏好**往往比几十万条低质的好。先求质量再求量。

## 偏好数据的"信号"自检

收集后，先问 3 个问题再训：
1. **一致性**：标注者之间一致吗？
2. **清晰度**：chosen 比 rejected 好在哪，能说清吗？
3. **代表性**：覆盖了真实用法和边界吗？

> 答不上来就别训——先回去改数据。模型学不会模糊的数据。
