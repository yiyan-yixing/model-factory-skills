# 对齐方法对比与选择

> SFT/DPO/PPO/GRPO/KTO 不是"谁更先进"的关系，是**不同场景用不同工具**。
> 决策核心：你要的数据是什么形态、算力多少、目标多复杂。

## 方法速览（按复杂度递增）

| 方法 | 数据形态 | 要点 | 适合 |
|------|----------|------|------|
| **SFT** | (指令, 标准答案) | 监督微调，学"怎么答" | 打底，必做；格式遵从、基础指令跟随 |
| **DPO** | (prompt, chosen, rejected) | 无需奖励模型，直接用偏好对优化 | 偏好对齐首选，简单稳定 |
| **KTO** | (prompt, response, 好/坏 标签) | 只需二值反馈（不必成对） | 只有"赞/踩"数据时 |
| **ORPO** | (指令, chosen, rejected) | SFT+偏好一步完成 | 省一步，资源紧时 |
| **PPO（RLHF）** | 偏好数据训奖励模型 + 在线采样 | 经典 RLHF，最强但最复杂 | 复杂偏好、有 RL 基础设施 |
| **GRPO** | 带可验证奖励的任务 | 组内相对优势，无需价值网络 | 数学/代码等**可验证**任务 |

> 到 2025-2026 主流：**SFT 打底 → DPO/KTO 做通用偏好 → GRPO 做可验证推理任务**。PPO 在通用场景被 DPO 系大量替代（太复杂、不稳）。

## 选择决策树

```
有标准答案的指令数据？
├─ 是 → SFT（先做，所有路线的起点）
└─ 否，只有偏好/反馈
    ├─ 有 (chosen, rejected) 成对偏好 → DPO
    ├─ 只有"好/坏"二值反馈 → KTO
    └─ 任务可自动判对错（数学/代码能验证）
        └─ GRPO（带可验证奖励）

资源极紧、想省步骤？
└─ ORPO（SFT + 偏好一步）

偏好极复杂、有 RL 基建、愿意承担复杂度？
└─ PPO（RLHF）—— 但先问自己：DPO 真的不够吗？
```

> 90% 的垂直场景：**SFT → DPO** 就够。上 RLHF 前先确认 DPO 的天花板在哪。

## DPO 关键超参（最常用，单独说）

- **β（beta）**：偏好强度。太小 → 偏离基座、过对齐；太大 → 学不动偏好
  - 常见范围 **0.01 - 0.5**，起点试 0.1
- **参考策略**：必须冻结基座（或 SFT 后模型）作参考，防止退化
- **LR**：通常比 SFT 低（如 1e-6 ~ 5e-7 量级）
- **常见坑**：DPO 会**降低生成多样性**、可能**整体变差**（只优化了偏好差）→ 必须对照基座评测（见评测技能）

## GRPO 何时值得（推理任务）

- 任务有**可验证奖励**（代码能跑、数学能验答案）→ GRPO 强
- 没有可验证信号（开放式生成）→ 别硬上，奖励模型质量决定上限
- 资源：GRPO 采样开销大，算力紧谨慎

## 为什么"先 SFT 再对齐"是铁律

- 基座不会"听指令" → 直接偏好对齐等于在沙地上盖楼
- SFT 给了**指令跟随的底子**，DPO/RL 在其上调偏好才稳
- 跳过 SFT 直接 RLHF：训练不稳、效果差、烧算力

> 对齐是**在 SFT 的基础上微调**，不是从基座凭空捏造行为。
