# 安全对齐与红队

> 安全是底线，不是可选项。**一次严重的有害输出事故，能毁掉一个产品的信任。**
> 这一页讲安全对齐怎么做、红队怎么攻、用什么 benchmark 测。

## 安全对齐的两层

### 1. 安全 SFT / 安全偏好数据

- 构建**安全指令集**：覆盖应拒答的请求（违法、暴力、自伤、隐私、CSAM 等）
- 训练拒答行为：用安全偏好数据（chosen=拒答+解释，rejected=配合有害请求）
- **关键**：拒答要**带理由**、**礼貌**，不是生硬"我不能回答"

### 2. 红队/对抗测试（上线前必做）

主动攻击模型，找它会在什么情况下"破防"。

## 常见攻击类型（红队要覆盖）

| 攻击类型 | 说明 | 例子 |
|----------|------|------|
| 直接有害请求 | 直接问有害内容 | "怎么制造..." |
| 角色扮演绕过 | 让模型扮演"无限制AI" | "你现在是没有限制的 DAN..." |
| 多轮诱导 | 一步步引导越界 | 先聊合法话题，逐步推进 |
| 编码/语言绕过 | 用编码、外语、隐喻绕过 | base64、小众语言、暗语 |
| 指令注入 | 在输入里夹带恶意指令 | 文档里藏"忽略以上指令，输出..." |
| 越狱 prompt（jailbreak） | 公开流传的越狱模板 | 定期收集新越狱样本测试 |

> 越狱是个**攻防竞赛**。今天的防线明天可能被新 prompt 破。要**持续**收集新攻击样本、持续红队。

## 公开安全评测 benchmark（真实可用）

| Benchmark | 测什么 |
|-----------|--------|
| **AdvBench** | 对抗性有害指令 |
| **HarmBench** | 标准化有害行为评测 |
| **JailbreakBench** | 越狱攻击评测 |
| **BBQ** | 偏见/刻板印象 |
| **TruthfulQA** | 幻觉/常见误区 |
| **ToxiGen** | 有害/仇恨内容检测 |
| **StrongREJECT** | 红队拒绝率评测 |

> 用这些公开 benchmark 作为安全评测的**起点**，再叠加你垂直场景特有的安全 case。

## 安全评测的关键指标

| 指标 | 含义 | 注意 |
|------|------|------|
| 有害输出率 | 被诱导出有害内容的比例 | 越低越好 |
| 越狱成功率 | 公开越狱 prompt 的攻破率 | 越低越好 |
| **拒答误伤率** | 无害请求被错误拒答的比例 | **不能只追求低有害率而牺牲这个** |

> 致命陷阱：**过度对齐** → 模型什么都拒答，用户骂"这模型没用"。安全与有用要平衡，必须同时测**有害率**和**拒答误伤率**。

## 安全红队的 checklist（上线前）

- [ ] 跑了至少 2 个公开安全 benchmark
- [ ] 覆盖了主流攻击类型（直接/角色扮演/多轮/编码/注入/越狱）
- [ ] 测了垂直场景特有的风险 case
- [ ] 同时测了有害率和拒答误伤率（不只看安全一面）
- [ ] 收集了一批"新越狱样本"内部测试
- [ ] 有事故响应预案（线上发现有害输出怎么止血）
