# 模型能力 PRD：代码补全 API

> 产出依据：`skills/model-product-prd/SKILL.md` 五步法。
> 版本：v0.1（草案，待过数据/评测质疑闸门）
> 时间盒：45min

---

## Step 1：能力与用户定义

```
能力名：单文件上下文代码补全（single-file context code completion）
        —— 给定光标位置前的代码，续写下一段最可能的代码。
目标用户：
  - 主用户：工具开发者 —— IDE 插件 / 编辑器扩展 / 内部 DevOps 平台的集成方
  - 终端受益者：在 VS Code / JetBrains / 自研 Web IDE 里写后端业务代码的中级工程师
核心场景：
  1. 用户在编辑器里停顿 ≥ 300ms，触发当前文件的光标续写（inline ghost text）
  2. 用户刚敲完函数签名/注释，按 Tab 接受一段函数体补全
  3. 工具方通过 REST/Streaming API 把 prefix（光标前文本）+ 少量后缀（光标后文本）+ 语言标签发给本能力，拿到 1 条补全建议
现有替代：
  - GitHub Copilot（贵、数据不出域顾虑、对内网私有框架支持差）
  - Tabnine（离线版效果一般）
  - 公司内部规则补全（正则/模板，仅覆盖固定 snippet）
  - 痛点：对内部框架、内部命名规范、内部 RPC 接口几乎无补全；通用模型补全又常补出公司不存在的库
```

> 能力不是"做一个更聪明的模型"，而是"给定单文件光标上下文，续写下一段可被 Tab 接受的代码"。MVP 明确排除跨文件、多行重构、对话式生成。

## Step 2：可评测的成功标准

> 每个维度都有数字 + 当前基线。无数字目标 = 训练完无法验收。

| 能力维度 | 评测目标 | 当前基线（参考） | 测什么 |
|---|---|---|---|
| 接受率 Accept Rate | **≥ 28%** | 内部规则补全 ~9% / Copilot ~30%（公开口径） | 用户 Tab 接受次数 / 展示次数，线上埋点 |
| 完全匹配 Exact Match（单行/多行评测集） | **≥ 42%** | 规则补全 ~12% | 5000 条内部代码评测集，prefix→ground truth 续写 |
| 编辑后接受率 Accepted-After-Edit | **≥ 12%** | 规则 ~3% | 接受但有修改的占比，衡量"接近可用" |
| 首字延迟 TTFT | **P95 ≤ 200ms** | Copilot ~250ms | 流式首 token 时间 |
| 全量延迟（典型 30 token 补全） | **P95 ≤ 800ms** | — | 服务端完成时间 |
| 格式遵从（仅输出代码、无解释/无 markdown fence） | **≥ 99.5%** | — | 评测集中格式违规样本比例 |
| 语言覆盖 | Python / Java / TypeScript / Go 四语种各自 Exact Match ≥ 35% | — | 分语种评测集 |
| 安全合规 | 不补全出内部不存在的库/接口比例 ≥ 95% | 规则补全 100%（白名单） | 评测集 + 抽样人工 |

> "模型效果更好"不可验收。"接受率 ≥ 28%、P95 延迟 ≤ 800ms"可验收。无数字目标打回。

## Step 3：已知边界与失败模式

```
已知边界：
- 上下文窗口仅限「当前单文件光标前后」+ 元信息（语言、文件路径、repo 根名），不做跨文件检索。
  理由：MVP 不上 RAG，跨文件需要单独能力（依赖图、向量库），属于 PRD 之外。
- 支持语种：Python / Java / TypeScript / Go。C++/Rust/SQL/Shell 不在 MVP。
- 最大 prefix 长度 8000 token，超出按尾部截断（保留光标附近上下文）。
- 单次返回最多 1 条候选（不做 multi-candidate UI），长度上限 64 token。

失败模式（必须向集成方文档化）：
- 输入过长 → 截断后可能丢失文件顶部 import/类型声明，导致补全错类型。
- 光标位于大型函数中部、且无类型注解 → 猜类型，Exact Match 显著下降。
- 内部私有框架（无公开文档）→ 依赖训练数据是否纳入，未纳入则可能补出外部相似库。
- 光标紧邻测试文件 / mock 代码 → 补全偏向 mock 风格，可能不符合业务代码规范。
- 大段重复样板代码（如 DTO/POJO）→ 可能整段复制附近相似块，含脏字段。
- 实时性差：API 不感知"用户 1 分钟前刚定义的、同一会话内但另一文件的符号"。
```

> 边界也是 `@evaluation` 的重点：评测集必须覆盖上述每个失败模式，验证其退化程度是否可接受。

## Step 4：MVP 范围

```
MVP 包含（1-3 个核心能力）：
  1. 单文件光标续写（FIM：prefix + suffix → middle 补全），流式返回 1 条候选
  2. 四语种支持（Python/Java/TypeScript/Go）
  3. REST + SSE 流式 API，带语言/文件路径输入

MVP 不含（明确砍掉）：
  - 跨文件/跨仓库上下文检索（RAG、依赖图）—— 下一期
  - 多候选排序（n-best）
  - 对话式代码生成 / 自然语言→代码
  - 代码重构、命名建议、注释生成 —— 这些是不同的能力，各自需要独立 PRD
  - 自动 import 补全
  - C++/Rust/SQL/Shell 等语种
  - 端侧/本地部署（MVP 仅服务端 API）
  - 私有数据微调（MVP 用通用底座 + prompt 工程；私有框架微调是 PRD v2）
```

> 砍掉"顺便加的"：内部框架微调、跨文件检索都是高频被塞进 MVP 的东西，本期一律不做，避免验收范围爆炸。

## Step 5：触发质疑闸门（必须过两道）

### 闸门 A — `@data-strategy` 质疑：数据可得性

```
质疑项 1：评测集 5000 条从哪来？
  → 内部代码仓脱敏抽取（去密钥/去 PII）+ 人工抽检 500 条 ground truth。
  风险：内部仓许可是否允许用于评测？需法务确认。
质疑项 2：训练数据是否包含内部私有框架（决定"不补出不存在库"能否达成）？
  → MVP 决定用通用底座，因此"内部框架支持"本身就是 MVP 边界外的目标。
  → 需把"对内部框架的 Exact Match"从 Step 2 主目标里降级为观测指标，不作为验收硬指标。
质疑项 3：延迟基线从哪测？
  → 需在目标硬件（A10/A100）上跑一次 benchmark 才能填基线，否则 200ms/800ms 是拍脑袋。
```

**结论：PRD 需修订** —— Step 2 中"内部框架 Exact Match"挪出硬指标；延迟基线标 TBC，等 benchmark。

### 闸门 B — `@evaluation` 质疑：评估可量化

```
质疑项 1：接受率 ≥ 28% 这个数怎么定的？
  → 对标 Copilot 公开口径 ~30%，但 Copilot 含跨文件。MVP 仅单文件，28% 可能偏高。
  → 建议 MVP 验收线降到 ≥ 22%，预留跨文件上线后再上调。需 PO 确认。
质疑项 2：Exact Match 对"多行函数体"过于严苛，是否补半行就判失败？
  → 引入分级评测：单行 EM / 多行 EM / line-level partial match（编辑距离 < 阈值）。
  → Step 2 的 42% 应拆成两个数。
质疑项 3：格式遵从 99.5% 怎么测？是否包括 SSE 协议层错误？
  → 拆两层：模型输出格式违规率 + API 协议层错误率，分开报。
```

**结论：PRD 需二次修订** —— 评测维度需细化分级。
