## 1. 能力定义

```
能力名：中文短文本情感分类
目标用户：需要批量分析用户评论/反馈情感倾向的开发者和产品团队
核心场景：自动判断一段中文短文本（如商品评论、用户反馈）是正面还是负面
现有替代：调用通用大模型 API 做分类，成本高、延迟大、对简单分类任务来说杀鸡用牛刀
```

**用户痛点详解**：

- **成本高**：调用 GPT-4/Claude 等通用大模型做简单情感分类，单次调用成本约 $0.01-0.03，批量处理 10 万条评论需 $1000-3000，而专用小模型推理成本可低至 $0.0001/条
- **延迟大**：通用大模型 P95 延迟通常 500ms-2s，对实时场景（如客服情绪监控）不可接受
- **杀鸡用牛刀**：二分类任务不需要通用推理能力，大模型的生成式能力反而是浪费
- **依赖外部服务**：通用 API 调用存在网络依赖、供应商锁定、限流等风险

**为什么 0.5B 微调模型能解决这个问题**：情感二分类是典型的窄任务，0.5B 参数的中文开源基座经 SFT/DPO 微调后足以捕捉中文短文本的情感特征。独立部署的微调模型消除了上述所有痛点。

## 2. 基座选型

| 项目 | 内容 |
|------|------|
| **选型** | Qwen2.5-0.5B 作为基座 |
| **选型理由** | 最小可用的中文开源预训练模型，支持 SFT/DPO 微调，社区活跃，文档完善 |
| **约束** | 0.5B 参数意味着推理资源需求高于原 20M 方案，但仍在单 GPU 可部署范围内（INT8 量化后约 500MB 显存占用） |
| **与 D6 对齐** | 采用开源基座微调路线，与公司决策 D6（开源基座微调）完全一致 |

**选型说明**：原计划 20M 参数在开源社区没有现成中文预训练基座。20M 不是标准尺寸，从更大模型蒸馏需要额外流程（且蒸馏在"不做清单"中），从零预训练的工作量超出流程验证时间预算。Qwen2.5-0.5B 是目前可找到的最小中文开源基座，参数量级适合微调而非从零训练，且社区生态成熟。

## 3. 可评测的成功标准

本 PRD 有两套成功标准，**分别独立判定**。流程走通是核心目标，模型效果是及格线。

### 3A. 流程走通标准（核心！）

每个环节必须有明确产出物，且产出物之间可追溯：

| 环节 | 必须产出 | 验收条件 | 状态 |
|------|----------|----------|------|
| 数据采集 | 原始数据集 + 清洗后数据集 v1 + 质量校验报告 | 校验报告通过（空值率 < 5%、标签分布偏差 < 10%、合成数据与真实数据分布偏差 < 15%） | |
| 数据策略 | 数据需求文档 + 来源清单 + 标注规范 | @ml-trainer 确认可训练 | |
| 预训练/微调 | checkpoint + 训练日志 + 实验记录(W&B) | loss 下降、模型能推理出结果 | |
| SFT 对齐 | SFT 数据集 + 对齐后模型 + 变更说明 | 输出格式正确（JSON） | |
| DPO 对齐 | 偏好数据 + DPO 后模型 | 无格式退化、无安全退化 | |
| 评测 | 评测报告 + Go/No-Go 判定 + 缺陷清单 | 评测完整执行，报告含"所以呢"结论 | |
| 推理优化 | 推理服务 + 量化前后 benchmark 对比 | 延迟达标（见 3B） | |
| API 封装 | API 规范 + 鉴权 + 健康检查端点 | curl 可调用、鉴权生效 | |
| 部署 | 在线 URL + Docker 镜像 + 监控配置 | 健康检查返回 200 | |
| QA 验证 | QA 测试报告 + Go/No-Go | 全用例通过 | |
| 反馈闭环 | 至少 1 轮 No-Go -> 重训 -> 再评测 的完整记录 | 回流路径跑通 | |

**3A 验收**：所有环节产出齐全 = 流程验证成功（核心目标达成）。

### 3B. 模型效果标准（及格线，不是 SOTA）

| 能力维度 | 评测目标 | 当前基线 | 达标判定 |
|----------|----------|----------|----------|
| 准确率 | >= 85%（二分类） | 见下方基线对照 | 必须 |
| F1 分数 | >= 0.83 | 见下方基线对照 | 必须 |
| 精确率 | >= 0.83 | 见下方基线对照 | 参考 |
| 召回率 | >= 0.83 | 见下方基线对照 | 参考 |
| 格式遵从 | 100%（输出必须是合法 JSON） | -- | 必须 |
| 有害输入拒答率 | >= 95%（通过），>= 99%（优秀），100%（理想目标） | -- | 必须 |
| P95 延迟（未量化） | <= 100ms | -- | 必须 |
| P95 延迟（INT8 量化） | <= 50ms | -- | 必须 |

**对照基线**：

| 基线 | 方法 | 预估准确率 | 说明 |
|------|------|------------|------|
| (a) 规则基线 | 正面词表匹配 | ~65-70% | 不需要 GPU 的传统方法，模型必须显著超越此基线才有存在价值 |
| (b) 通用 API 基线 | GPT-4o-mini / Claude Haiku 等轻量 API | ~88-92% | 效果好但成本高、延迟大，0.5B 模型需在成本/延迟维度有数量级优势 |
| (c) 同级开源模型基线 | 其他 0.5B 级别中文模型在相同任务上的表现 | 待评测 | 如适用，用于横向对比 |

**评测目标（相对基线优势判定）**：

- 必须**显著超越**规则基线（统计显著，p < 0.05）
- 在成本/延迟维度相比通用 API 有数量级优势（成本 < 1/10，延迟 < 1/5）
- 准确率绝对值目标 >= 85%（0.5B 模型应比原 20M 目标更高）
- F1 绝对值目标 >= 0.83

**Go/No-Go 判定规则**：

- **Go**：显著超越规则基线（p < 0.05） + 成本/延迟相比通用 API 有数量级优势（成本 < 1/10，延迟 < 1/5） + 所有"必须"项达标
- **No-Go**：未显著超越规则基线，或成本/延迟无数量级优势，或任一"必须"项不达标

### 3A/3B 关系与判定

**3A 和 3B 分别独立判定，不互相依赖。**

- **3A 通过 + 3B 通过** = 流程验证完整成功
- **3A 通过 + 3B 未达标** = 流程验证成功但模型不满足交付标准，记录经验教训
- **3A 未通过** = 流程验证失败（无论 3B 结果如何）

**3B 未达标时的决策树**：

| 3B 差距 | 判定条件 | 行动 |
|----------|----------|------|
| 差距小 | F1 差目标 0.03 以内 | 补充数据重训 1 轮 |
| 差距中 | F1 差目标 0.03-0.10 | 诊断数据/训练/基座瓶颈，写分析报告，决定是否继续 |
| 差距大 | F1 差目标 0.10 以上 | 基座选型或流程设计有问题，流程验证部分失败，需要重新审视 |

**3B 未达标但 3A 通过时**：流程验证成功但模型不满足交付标准，将分析报告和经验教训写入 `blackboard/`，为后续迭代提供依据。

## 4. 已知边界与失败模式

0.5B 参数模型有明确的物理限制，诚实声明，不隐瞒：

### 已知边界

- **不支持长文本**：超过 512 token 的输入会被截断，截断后情感判断可能不完整
- **不支持中性分类**：只做正面/负面二分类，中性文本会被强制归入正面或负面
- **对讽刺/反语理解有限**：0.5B 参数对"这手机真是太好了，摔一次就碎"的讽刺语义理解仍有限
- **跨领域迁移能力弱**：训练数据以商品评论为主，迁移到金融/医疗等领域效果可能退化
- **输入为空或纯符号时可能误判**：极端输入无法保证合理输出
- **中英混合文本效果可能下降**：训练数据以纯中文为主，中英混合（如"这个 app 不好用"）可能影响判断

### 失败模式

| 失败模式 | 触发条件 | 预期表现 | 缓解措施 |
|----------|----------|----------|----------|
| 讽刺误判 | 输入含反语/讽刺 | 正面文本被判负面或相反 | 置信度 < 0.6 时输出 low_confidence 标记 |
| 长文本截断 | 输入 > 512 token | 截断后情感可能翻转 | API 返回 truncated 警告 |
| 领域外退化 | 非商品评论领域 | 准确率可能低于 85% | API 文档声明适用领域 |
| 对抗输入 | 蓄意构造的对抗样本 | 可能被绕过 | 红队测试覆盖 |
| 格式崩溃 | 极端输入 | 输出非 JSON | 后处理兜底 + 重试 |

## 5. MVP 范围

```
MVP 包含：
1. 二分类情感判断（正面/负面）+ 置信度分数
2. 中文短文本输入（<= 128 字），JSON 输出
3. 单条推理 API（HTTP POST）

MVP 不含：
- 中性分类
- 多标签情感
- 方面级情感分析（ABSA）
- 长文本支持（> 128 字）
- 多语言支持
- 批量推理优化（单条推理即可）
- 流式输出（分类不需要）
- 自定义模型微调接口
- 模型版本管理 API
- A/B 实验框架
```

### API 输出格式

```json
{
  "label": "positive" | "negative",
  "confidence": 0.92,
  "model_version": "sentiment-0.5b-v1",
  "warnings": []
}
```

`warnings` 数组在以下情况非空：
- 输入被截断：`{"type": "truncated", "original_length": 256, "truncated_to": 128}`
- 置信度过低：`{"type": "low_confidence", "confidence": 0.52}`

## 6. 数据需求

### 6.1 训练数据

| 数据集 | 规模 | 来源 | 用途 | 备注 |
|--------|------|------|------|------|
| ChnSentiCorp | ~12K 条 | 公开数据集（酒店/书籍/电脑评论） | 主训练集 | 已标注正面/负面 |
| 合成增强数据 | ~8K 条 | 模板生成 + 回译增强 | 补充训练集 | 目标总训练数据 20K 条 |

**合成增强策略**：
- 同义词替换：随机替换非情感关键词，保持标签不变
- 回译增强：中文 -> 英文 -> 中文，增加表达多样性
- 情感关键词模板：按领域生成含典型情感词的短文本

### 6.2 评测数据

| 数据集 | 规模 | 来源 | 用途 | 备注 |
|--------|------|------|------|------|
| 评测集 | 2K 条 | 从 ChnSentiCorp 独立切分 | 评测 | 必须与训练数据零重叠 |
| 对抗测试集 | 100 条 | 人工构造 | 安全评测 | 讽刺/对抗/边界样本 |

**切分规则**：按 document 级别切分（非 sentence 级别），确保同一评论的不同句子不会分跨训练和评测集。按领域分层采样，保证评测集领域分布与训练集一致。

### 6.3 SFT 数据

| 数据集 | 规模 | 格式 | 用途 |
|--------|------|------|------|
| SFT 指令数据 | 1K 条 | instruction -> JSON 输出 | 格式对齐 |

SFT 数据示例：
```json
{
  "instruction": "请判断以下文本的情感倾向，以 JSON 格式输出。",
  "input": "这家酒店服务态度很好，房间也干净整洁",
  "output": "{\"label\": \"positive\", \"confidence\": 0.95}"
}
```

### 6.4 偏好对齐数据

| 数据集 | 规模 | 格式 | 用途 |
|--------|------|------|------|
| DPO 偏好数据 | 500 条 | chosen/rejected 对 | DPO 对齐 |

chosen/rejected 示例：
- chosen：格式正确 + 情感判断准确
- rejected：格式错误 / 判断错误 / 置信度校准差

### 6.5 安全红队数据

| 数据集 | 规模 | 用途 |
|--------|------|------|
| 红队测试用例 | 100 条 | 有害输入拒答验证 |

红队用例类型：仇恨言论输入、指令注入尝试、隐私信息提取尝试、非法内容生成诱导。

### 6.6 数据版本化

- 工具：DVC 或 git-lfs
- 命名规范：`data/{type}/v{N}`，如 `data/train/v1`、`data/eval/v1`
- 每个版本附元数据文件：来源、规模、切分规则、质量校验结果
- 版本间变更必须记录 diff

## 8. 流程走通验收标准

按环节列出每个环节必须产出的 artifact，以及负责角色：

| 环节 | 必须产出 | 验收方式 | 负责角色 |

| 数据采集 | 原始数据集 + 清洗后数据集 v1 + 质量校验报告 | 校验报告通过（空值率 < 5%、标签偏差 < 10%、合成数据与真实数据分布偏差 < 15%） | @data-engineer |
| 数据策略 | 数据需求文档 + 来源清单 + 标注规范 | @ml-trainer 确认可训练 | @data-strategy |
| 微调 | checkpoint + 训练日志 + 实验记录(W&B) | loss 下降、能推理 | @ml-trainer |
| SFT 对齐 | SFT 数据集 + 对齐后模型 + 变更说明 | 输出格式正确（JSON） | @ml-alignment |
| DPO 对齐 | 偏好数据 + DPO 后模型 | 无格式退化、无安全退化 | @ml-alignment |
| 评测 | 评测报告 + Go/No-Go 判定 + 缺陷清单 | 评测完整执行 | @evaluation |
| 推理优化 | 推理服务 + 量化前后 benchmark 对比 | 延迟达标 | @ml-serving |
| API 封装 | API 规范 + 鉴权 + 健康检查端点 | curl 可调用 | @backend |
| 部署 | 在线 URL + Docker 镜像 + 监控配置 | 健康检查 200 | @infra |
| QA 验证 | QA 测试报告 + Go/No-Go | 全用例通过 | @evaluation |
| 反馈闭环 | 至少 1 轮 No-Go->重训->再评测 的完整记录 | 回流路径跑通 | @evaluation + @ml-trainer |

### 反馈闭环验证

流程验证的关键是证明"不行时能回来"。必须跑通至少 1 轮：

```
@evaluation 报告 No-Go（如准确率 82%，不达标）
  -> 缺陷清单写入 blackboard
  -> @ml-trainer 根据缺陷分析调整（如补充领域数据、调整学习率）
  -> 重新训练 -> 重新评测 -> 准确率提升到 86%
  -> @evaluation 报告 Go
  -> 记录：No-Go 原因 -> 修正措施 -> 结果，完整链路可追溯
```

如果首轮评测就 Go，则人为构造一次 No-Go 场景（如使用更严格的阈值或对抗样本），强制触发回流，验证回流路径通畅。

### 整体验收

**3A（流程走通）和 3B（模型效果）分别独立判定：**

- **3A 通过 + 3B 通过** = 流程验证完整成功
- **3A 通过 + 3B 未达标** = 流程验证成功但模型不满足交付标准，按 3B 决策树处理，记录经验教训
- **3A 未通过** = 流程验证失败

3B 未达标时，按决策树判定差距等级并采取对应行动（差距小→补充数据重训；差距中→诊断瓶颈写分析报告；差距大→重新审视基座选型或流程设计）。

验收由 @evaluation 执行，@po 确认。验收通过后写入 `blackboard/current-sprint.md`。

## 不做清单（防范围蔓延）

> 以下是明确不做的事项。任何人提出"顺便加上"的需求，引用此清单拒绝。

| 编号 | 不做 | 拒绝理由 |
|------|------|----------|
| X1 | 中性分类 | MVP 边界：二分类，加中性是三分类，是另一个模型 |
| X2 | 多标签情感（如同时开心+感动） | 超出二分类 MVP 范围 |
| X3 | 方面级情感分析（ABSA） | 复杂度远超二分类，需要完全不同的模型架构和数据 |
| X4 | 长文本支持（> 128 字） | 0.5B 模型上下文窗口有限，长文本需要更大模型 |
| X5 | 多语言支持 | 训练数据只有中文，跨语言需要多语言预训练 |
| X6 | 批量推理优化 | 分类场景批量可由调用方并发实现，模型侧不优化 |
| X7 | 流式输出 | 分类结果是单次输出，无需流式 |
| X8 | 自定义模型微调接口 | 流程验证阶段不做用户定制 |
| X9 | 模型版本管理 API | 流程验证阶段不做运维功能 |
| X10 | A/B 实验框架 | 流程验证阶段不做实验平台 |
| X11 | 模型蒸馏到更小规模 | 流程验证只验证 0.5B 这一个规模 |
| X12 | 多模型集成/投票 | 流程验证只验证单模型 |
| X13 | 实时监控仪表盘 | 流程验证只做健康检查，不做可视化 |
| X14 | 自动数据飞轮（线上数据回流训练） | 流程验证只做人工回流，不做自动闭环 |
| X15 | 非 REST 协议（gRPC/WebSocket） | 流程验证只做 HTTP REST |
