## 走查要点

### 1. 方向一致性：通过

PRD 与公司战略决策（D1-D6）基本对齐：
- D1 垂直场景：情感分类是垂直任务，且 PRD 清醒地定位为"流程验证而非产品交付"，不越界。
- D2 评测先行：PRD 包含量化评测指标和 Go/No-Go 判定机制，方向正确（但评测设计本身有问题，见要点3）。
- D3 API/调用优先：MVP 包含单条推理 API，对齐。
- D4 数据壁垒：用公开数据做流程验证可以理解，不做数据壁垒验证不阻断。
- D6 开源基座微调：明确走微调路线，对齐。

**一个注意点**：流程验证完成后应产出 Sprint-1 的垂直场景候选清单和商业化假设，否则 Sprint-0 验证的流程没有下一步方向。这是改进项，不阻断，但建议 PRD 附录中补充。

### 2. 基座选型：不通过（阻断）

PRD 写了"20M 参数"但完全没有回答**从哪来**：

- 目前开源社区 20M 级别的中文预训练模型几乎不存在。Qwen2.5-0.5B 是 500M，再往下没有现成中文基座。
- 如果选微调路线（D6），需要明确微调哪个基座。20M 不是标准尺寸，大概率需要从更大模型蒸馏而来。
- 但蒸馏在"不做清单"中（X11），这意味着蒸馏步骤被排除在流程之外。
- 如果退而求其次从零预训练一个 20M 中文基座，那本质上就是自训基座（与 D6 矛盾），且预训练本身的工作量可能超过流程验证的时间预算。

**结论**：20M 这个数字悬在空中，没有落地路径。这不是小问题，是可执行性的根基。

**修改要求**：
- (a) 明确基座来源方案：微调哪个开源基座？如果是蒸馏，需从"不做清单"中移除蒸馏；如果是预训练，需重新评估时间预算。
- (b) 如果 20M 基座确实不可得，将参数规模调整到可找到开源基座的范围（如 0.5B / Qwen2.5-0.5B），并相应调整模型效果标准。
- (c) 无论选择哪种方案，必须在 PRD 中写明基座选型的理由和约束。

### 3. 评测标准：不通过（阻断）

两个问题：

**3.1 评测基线缺失**

PRD 给出了绝对值目标（准确率 >= 80%、F1 >= 0.78），但没有给出对照基线。80% 准确率在真空中没有意义——

- 规则方法（正面词表匹配）大概 65-70%
- 通用大模型 API（GPT-4o-mini）大概 88-92%
- 80% 比规则好，但比通用 API 差。那客户为什么不用通用 API？

D2 决策是"评测先行"，评测的核心价值不在数字本身，而在**数字如何指导决策**。没有基线对照的评测只是自嗨——恰恰是 D2 要防范的。

**修改要求**：
- 补充至少三个基线：(a) 规则基线（词表匹配） (b) 通用大模型 API 基线 (c) 同级别开源模型基线（如适用）
- Go/No-Go 标准调整为：必须显著超越规则基线（即 80% 准确率必须比词袋/词表方法有统计显著优势），且在成本/延迟维度相比通用 API 有数量级优势

**3.2 2A/2B 关系模糊**

流程走通标准（2A）和模型效果标准（2B）的关系是什么？

- 如果流程走通但模型效果不达标（比如 F1 只有 0.72），算不算流程验证通过？
- 这是"流程验证"项目的核心问题，不回答它，项目会卡在判定环节。

**修改要求**：
- 明确 2A 和 2B 分别独立判定
- 2A 通过 = 流程验证成功（核心目标达成）
- 2B 未达标时，必须给出"所以呢"：是数据问题？基座问题？还是流程设计本身有问题？PRD 应给出决策树
- 2A 和 2B 同时通过 = 流程验证完整成功

### 4. 可执行性：有条件通过

**时间预算**：3 天过于乐观，5 天可行但紧凑。建议调整为 4-6 个工作日，或缩小 MVP 范围到"数据->训练->评测->API"四环节走通（3-5 天可行），部署和反馈闭环放到后续迭代。这是改进项，不阻断。

**数据规模**：20K 训练数据训练 20M 模型，数据量是参数量 1000 倍，足够。通过。

**合成数据质量**：PRD 要求了合成增强但没有要求合成数据的质量验证步骤。建议在数据采集环节的验收条件中增加"合成数据与真实数据分布偏差 < 15%"。改进项，不阻断。

**有害输入拒答率 100%**：实际中几乎不可能保证，作为硬性 Go/No-Go 标准可能导致项目永远 No-Go。建议设为：>= 95% 通过，>= 99% 优秀，100% 为理想目标。改进项，不阻断。
