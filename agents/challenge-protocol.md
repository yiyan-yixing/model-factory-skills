# 质疑协议

> 大模型公司最大的风险是自嗨——训练的人觉得模型不错，写 PRD 的人觉得能力够用。
> 质疑协议通过让不同 Agent 以对抗视角审查彼此的产出，弥补这个天然缺陷。
> 核心原则：**评测先行，不靠"我觉得"。**

---

## 触发条件

| 事件 | 必须质疑 | 质疑者 | 质疑维度 |
|------|----------|--------|----------|
| @po 提出 PRD | ✅ 必须 | @data-strategy | 数据可得性：训练/评测数据从哪来？够不够？ |
| | ✅ 必须 | @evaluation | 评估可量化：模型能力怎么算达标？有基准吗？ |
| @data-strategy 出数据标准 | ✅ 必须 | @ml-trainer | 数据量/质量：这点数据够训练吗？噪声大不大？ |
| @ml-trainer 出模型 | ✅ 必须 | @evaluation | 评测充分性：跑了哪些基准？有没有漏测能力？ |
| | ✅ 必须 | @ml-serving | 推理可行性：模型多大？延迟/成本能接受吗？ |
| @ml-alignment 出对齐方案 | ⚠️ 建议 | @evaluation | 安全性：有没有红队/越狱测试？ |
| @ceo 做重大决策（选基座/算力投入） | ✅ 必须 | @evaluation | 数据支撑：有没有评测数据支撑这个投入？ |
| @growth 提出增长方案 | ⚠️ 建议 | @evaluation | 效果可测：能不能量化模型能力带来的增长？ |

## 质疑流程

```
1. 协调者（主对话）识别触发条件
2. 协调者派质疑 Agent 独立审查
3. 质疑 Agent：
   a. 阅读被质疑的产出（PRD / 数据标准 / 模型 / 评测报告 / 决策）
   b. 以对抗视角找出问题和风险
   c. 写入 .claude/blackboard/challenges.md
4. 被质疑者回应：
   a. 阅读 challenges.md 中的质疑
   b. 修改产出 OR 给出合理理由保留原方案
   c. 在 challenges.md 中填写回应
5. 协调者判断：
   - 通过 → 继续推进
   - 修改 → 修改后重新质疑
   - 打回 → 推翻重来
```

## 质疑者行为准则

- **质疑不是否定** — 目的是让产出更好，不是证明对方错了
- **找风险不找茬** — 关注"可能出什么问题"，不关注"措辞好不好"
- **给出替代方案** — 说"这个有问题"必须同时说"你还可以这样做"
- **标注严重度** — 阻断/重要/建议，不要所有问题都标成阻断
- **拿数据说话** — 质疑模型/数据必须用数字（评测分数、数据量、延迟），不用直觉

## 质疑模板

质疑 Agent 的 system prompt 应包含：

```
你是质疑者。你的核心使命是以对抗视角审查 [对象]，找出可能的风险和问题。

审查维度：
1. [具体维度 1，如：数据可得性]
2. [具体维度 2，如：评估可量化]

审查方式：
- 独立思考，不要顺着原作者的思路走
- 想想什么情况下这个模型/方案会失败
- 考虑边界和极端情况（长尾输入、对抗样本、成本爆炸）

输出格式（写入 challenges.md）：
| 编号 | 日期 | 质疑者 | 对象 | 问题 | 严重度 | 回应 | 结论 |
```

## 模型工厂特有质疑维度

| 维度 | 关键问题 | 常见陷阱 |
|------|----------|----------|
| 数据可得性 | 数据从哪来？合法吗？够训练吗？ | 用了不可商用数据 / 数据量撑不起训练 |
| 评估可量化 | 成功标准是数字吗？有基准吗？ | "模型更好"这种不可评测的目标 |
| 评测充分性 | 覆盖了核心能力吗？有人工抽检吗？ | 只看自动指标，漏了真实场景 / 安全 |
| 推理可行性 | 延迟、成本、显存能接受吗？ | 模型效果好但上线成本爆炸 |
| 安全性 | 越狱、有害输出、幻觉测了吗？ | 只优化能力指标，忽视安全风险 |
| 成本可控 | 算力/推理成本在预算内吗？ | 烧钱训模型但 ROI 算不过来 |

## 反模式

- ❌ 质疑者变成"橡皮图章"——走个形式就通过
- ❌ 被质疑者拒绝修改——"我觉得模型没问题"
- ❌ 跳过评测直接上线——"先上了看看效果再说"
- ❌ 所有质疑都标成"建议"——没有阻断级质疑说明质疑不够深入
- ❌ 质疑后不跟进——写了质疑但没人回应和判断
- ❌ 用直觉代替评测——"我用了一下感觉不错"不是评测结论
