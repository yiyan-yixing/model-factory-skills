# 代码数据策略

> 代码大模型 = 代码数据驱动的模型。**代码数据的去重、质量、合规，决定模型上限。**
> 这一页讲代码数据从哪来、怎么洗、怎么合规、怎么造指令数据。

## 代码数据来源

| 来源 | 说明 | 注意 |
|------|------|------|
| **公开代码语料**（The Stack / 各种 GitHub 爬取集） | 预训练主力 | **协议过滤 + 去凭证**（见下） |
| GitHub 直采 | 自定义采集 | 遵守 ToS、API 限流 |
| 合成代码数据 | 模型生成代码/指令 | 需校验可运行性，防自循环 |
| 企业私有代码（微调用） | 差异化壁垒 | **数据不出域**，合规授权 |
| 代码指令数据 | 指令微调用 | 从代码构造指令（Evol-Instruct 思路） |

## 代码去重（比文本更关键）

代码重复极常见（fork、模板、boilerplate），去重要更狠：

| 粒度 | 方法 |
|------|------|
| **文件级精确/模糊去重** | MinHash，去 fork 和复制粘贴 |
| **函数/方法级去重** | 更细粒度，去 boilerplate |
| **跨仓库去重** | 同一库被多次包含 |

> 代码去重不彻底 → 模型背模板而非学语法，且评测虚高（HumanEval 等可能被近似重复污染）。

## 质量过滤

- **星标/质量信号**：高星仓库质量更高（启发式，非绝对）
- **可运行性**：能解析、语法正确（过滤坏代码）
- **去自动生成代码**：去 `package-lock.json`、minified、生成产物
- **长度/复杂度过滤**：过长/过短剔除
- **去注释/文档稀疏**：保留有上下文的代码

## 合规底线（代码场景特有，必须做）

| 合规项 | 说明 |
|--------|------|
| **开源协议过滤** | 按协议白名单过滤（避免 GPL/AGPL 等传染性协议引发商用问题）—— **法律雷** |
| **去凭证** | 清除 API key/密码/token/私钥（正则 + 检测）—— **安全事故雷** |
| **去 PII** | 代码注释里的个人信息 |
| **企业授权** | 私有代码必须有明确授权才能用于训练 |

> **训练数据含 GPL 代码或泄露的凭证，是代码大模型公司最现实的法律/安全风险。** 这一项做不好，公司可能直接挂。@data-strategy 的 KPI 必须含"协议合规 100% + 凭证清除 100%"。

## 多语言与配比

- 主流语言（Python/JS/Java/Go/...）占比要够
- 长尾语言靠合成 + 少量高质量
- 配比影响语言能力分布——按目标用户的技术栈调

## 代码指令数据构建

预训练给"代码能力"，**指令微调**给"听懂人的开发需求"：

- 从代码反向构造指令（"这段代码做什么" → 指令-代码对）
- Evol-Instruct 思路：进化指令复杂度，提升指令多样性
- 覆盖：补全/生成/解释/调试/重构/写测试/转语言
- 高质量人工精标关键场景（企业真实开发任务）

## 评测数据隔离（防污染）

- 代码 benchmark（HumanEval 等）**必须从训练数据中剔除**（含近似去重）
- 用**时间分窗 benchmark**（LiveCodeBench，按发布时间切）防新题泄露
- 建**私有代码评测集**（企业内部真实任务，不公开）

## 与其他角色衔接

- @data-strategy 定代码数据标准（协议白名单/质量门槛）→ @data-engineer 执行
- @ml-trainer 用合规清洗后的代码数据训练，**记录数据版本**
- @evaluation 用防污染的代码 benchmark 评测
